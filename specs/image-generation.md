# Feature Specification: Image Generation
**Status**: Implemented
**Created**: 2026-04-13

## Overview
Each pet has a unique visual appearance — a small creature generated by AI. Images are queued as background jobs and stored in MinIO/S3. The system supports two providers (OpenRouter and ComfyUI), 5 visual styles, and produces both static PNGs and animated GIFs from 3x2 sprite sheets.

## User Stories

### Pet gets a unique generated image on registration (Priority: P1)
When a pet is created, an image generation job is queued for all stages.
**Acceptance Scenarios**:
1. Given a newly created pet, When registration completes, Then an ImageGenerationJob is created with status=PENDING
2. Given the job queue processes the job, Then images are generated for all 6 stages and stored in MinIO
3. Given the provider is unavailable, Then the job is retried up to 3 times before marking FAILED

### Same repo always produces the same appearance (Priority: P1)
Appearance is deterministic from the repo identity hash.
**Acceptance Scenarios**:
1. Given owner="octocat" and repo="hello-world", When get_pet_appearance is called twice, Then the same color, body_type, feature, and seed are returned both times
2. Given two different repos, Then their appearances differ (probabilistically)

### Admin can trigger per-pet image regeneration (Priority: P2)
Admins can force regeneration of images for a specific pet from the admin panel.
**Acceptance Scenarios**:
1. Given an admin is logged in, When they click regenerate on a pet's admin page, Then a new ImageGenerationJob is created
2. Given the old images exist in storage, Then they are overwritten when the new job completes

### Animated GIF reflects current mood (Priority: P1)
The GIF served at the badge endpoint uses the pet's current mood to select the animation sequence.
**Acceptance Scenarios**:
1. Given mood == "dancing", When compose_animated_gif is called, Then it uses the HAPPY frame sequence [happy, idle, happy, idle]
2. Given mood == "lonely", Then it uses [sleeping, sad, sleeping]
3. Given health < 30 (regardless of mood), Then the "sick" animation overrides

## Functional Requirements
- **FR-001**: Two providers: `openrouter` (default, `google/gemini-2.5-flash-image`) and `comfyui`; switchable via `image_generation_provider` config
- **FR-002**: Image generation is always async — requests go to the job queue, not inline
- **FR-003**: Job queue: FIFO, MAX_ATTEMPTS=3, POLL_INTERVAL_SECONDS=10
- **FR-004**: Appearance attributes derived deterministically: color palette (16 options), body type (8 options), feature (10 options), seed — all via SHA-256 of `owner/repo`
- **FR-005**: 5 visual styles: kawaii (default), doom_metal, wizard, retro_scifi, minimalist
- **FR-006**: Sprite sheet layout: 3 columns x 2 rows = 6 frames (idle, blink, happy, sad, sick, sleeping)
- **FR-007**: Background removal: BFS flood-fill from all 4 corners, tolerance=40 per channel, regardless of exact AI-generated shade
- **FR-008**: GIF composition: mood-to-frame-sequence mapping; health < 30 forces sick animation
- **FR-009**: Frame durations (ms): idle=400, blink=180, happy=350, sad=500, sick=500, sleeping=700
- **FR-010**: Canonical appearance description stored per-pet for sprite sheet consistency across regenerations
- **FR-011**: Images stored with key pattern `{owner}_{repo}_{stage}.png` in MinIO bucket
- **FR-012**: `canonical_appearance` field on Pet model persists the description used for generation

## Technical Notes
- Key files: `src/github_tamagotchi/services/image_generation.py`, `src/github_tamagotchi/services/sprite_sheet.py`, `src/github_tamagotchi/services/openrouter.py`, `src/github_tamagotchi/services/image_queue.py`
- `get_pet_appearance(owner, repo)` returns `PetAppearance` dataclass
- `build_sprite_sheet_prompt()` generates a single prompt for the full 3x2 grid
- `extract_frames(sprite_sheet_bytes)` slices the grid, removes background, returns list of PNG bytes
- `compose_animated_gif(frames, mood, health)` selects sequence and encodes GIF with palette quantization
- GIF uses `disposal=2` (restore to background) and `loop=0` (infinite)
- Background removal uses top-left corner pixel as background color reference (not hardcoded magenta)

## Success Criteria
- SC-001: Every pet has at least one generated image visible on its profile page
- SC-002: Image generation failures do not prevent pet creation or polling
- SC-003: The same repo identity always produces the same visual appearance
- SC-004: Animated GIFs loop correctly and reflect the pet's current mood
- SC-005: All 6 sprite frames are extracted and stored per pet per style
